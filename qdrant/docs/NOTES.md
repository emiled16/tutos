# Qdrant Semantic Search & RAG — Theory & Notes

## 1. Vector Databases — Why They Exist

Traditional databases index structured data (rows, columns, B-trees). Vector databases index **high-dimensional vectors** — numerical representations of unstructured data (text, images, audio).

**The core operation:** given a query vector, find the K most similar vectors in a collection of millions. This is **Approximate Nearest Neighbor (ANN)** search — exact search is O(n) and too slow at scale.

## 2. Embedding Spaces and Similarity Metrics

### What Are Embeddings?
Dense numerical vectors (e.g., 384 or 768 dimensions) where **semantic similarity maps to geometric proximity**. "King" and "Queen" are close; "King" and "Banana" are far.

### Similarity Metrics

| Metric | Formula | Use Case |
|--------|---------|----------|
| **Cosine** | cos(θ) = A·B / (‖A‖·‖B‖) | Normalized embeddings; most common for text |
| **Dot Product** | A·B | When magnitude matters (popularity-weighted) |
| **Euclidean** | ‖A - B‖ | When absolute distance matters |

**For text search:** Use cosine similarity. Most sentence-transformer models produce normalized vectors, so cosine ≈ dot product.

## 3. HNSW Index

**Hierarchical Navigable Small World** — the dominant ANN index structure.

### How It Works
- Builds a multi-layer graph where each node is a vector
- Top layers have few nodes (long-range connections) for fast traversal
- Bottom layers have all nodes (short-range connections) for precision
- Search starts at the top and descends, getting more precise

### Key Parameters

| Parameter | What It Controls | Tradeoff |
|-----------|-----------------|----------|
| `m` | Max edges per node | Higher = better recall, more memory |
| `ef_construct` | Search width during build | Higher = better index quality, slower build |
| `ef` | Search width at query time | Higher = better recall, slower queries |

### Typical Values
- `m = 16` — good balance for most datasets
- `ef_construct = 100-200` — higher for production quality
- `ef = 128` — tune based on desired recall/latency tradeoff

## 4. Dense vs Sparse Vectors

### Dense Vectors
- Fixed-length (e.g., 768 dimensions), every element is non-zero
- Generated by neural models (sentence-transformers, OpenAI embeddings)
- Capture **semantic meaning** ("automobile" matches "car")
- Struggle with: exact keyword matching, rare terms, proper nouns

### Sparse Vectors
- Variable-length, most elements are zero
- Generated by BM25, TF-IDF, or learned models (SPLADE)
- Capture **lexical matching** (exact word overlap)
- Struggle with: synonyms, paraphrases, semantic similarity

### Why Combine Both?
Neither is strictly better. Dense excels at semantic similarity; sparse excels at keyword precision. Hybrid search combines both for the best of both worlds.

## 5. Hybrid Search and Fusion Strategies

### Reciprocal Rank Fusion (RRF)
Combines ranked lists from multiple retrieval methods:

```
RRF_score(d) = Σ 1 / (k + rank_i(d))
```

Where `k` is a constant (typically 60) and `rank_i(d)` is document d's rank in result list i.

**Why RRF works well:**
- Rank-based, so doesn't need score normalization between methods
- Robust to outlier scores
- Simple to implement

### Linear Combination
Weighted sum of normalized scores:

```
score(d) = α * dense_score(d) + (1 - α) * sparse_score(d)
```

Requires score normalization. `α` is a hyperparameter to tune.

## 6. Quantization for Storage Efficiency

### Scalar Quantization
Reduces each float32 dimension to int8 (4x memory reduction). Minimal quality loss.

### Product Quantization (PQ)
Groups dimensions into subspaces, quantizes each independently. Higher compression, more quality loss.

### Binary Quantization
Each dimension → 1 bit (32x compression). Works surprisingly well for high-dimensional vectors. Best for overparameterized models (>1024 dims).

## 7. Payload Indexing for Filtered Search

Qdrant supports filtering on metadata ("payloads") during vector search.

### Filter Types
```python
from qdrant_client.models import Filter, FieldCondition, MatchValue, Range

# Must match (AND)
Filter(must=[FieldCondition(key="category", match=MatchValue(value="science"))])

# Should match (OR)
Filter(should=[
    FieldCondition(key="year", range=Range(gte=2020)),
    FieldCondition(key="source", match=MatchValue(value="arxiv")),
])

# Must not match (NOT)
Filter(must_not=[FieldCondition(key="status", match=MatchValue(value="draft"))])
```

### Index Types
- **Keyword index** — exact match on string values
- **Integer/float index** — range queries on numeric values
- **Full-text index** — substring matching
- **Geo index** — location-based filtering

**Best practice:** Index fields you'll filter on frequently. Unindexed filters scan all payloads.

## 8. Collection Configuration

### Shard Count
- More shards = better write throughput and parallelism
- Fewer shards = lower overhead for small collections
- Rule of thumb: 1 shard per CPU core for large collections

### Replication Factor
- `1` = no redundancy (fine for development)
- `2+` = data replicated across nodes (production)

### Write Consistency
- `1` = write succeeds after 1 replica acknowledges
- `majority` = write succeeds after majority acknowledges
- `all` = write succeeds only after all replicas acknowledge

## 9. RAG Architecture

### The RAG Pipeline
```
Query → Retrieve relevant chunks → Build prompt with context → Generate answer → Return with citations
```

### Components
1. **Retriever** — finds the top-k most relevant document chunks
2. **Context builder** — formats retrieved chunks into an LLM prompt
3. **Generator** — produces an answer grounded in the provided context
4. **Citation tracker** — maps answer claims back to source chunks

### Why RAG Over Fine-Tuning?
- **Freshness** — knowledge can be updated by re-indexing (no retraining)
- **Grounding** — answers cite specific sources (less hallucination)
- **Cost** — cheaper than fine-tuning for domain adaptation
- **Control** — easily restrict which documents the model can reference

## 10. Chunking Strategies and Their Impact

### Fixed-Size Chunking
Split every N tokens/characters with optional overlap.
- **Pro:** Simple, predictable chunk sizes
- **Con:** May split sentences/paragraphs mid-thought

### Semantic Chunking
Split at natural boundaries (paragraphs, sections, topics).
- **Pro:** Each chunk is a coherent unit of meaning
- **Con:** Variable chunk sizes, more complex to implement

### Recursive Chunking
Try to split by paragraph, then sentence, then word — falling back to smaller units only when chunks are too large.
- **Pro:** Best balance of coherence and size
- **Con:** Most complex to implement

### Impact on Retrieval
- Too small chunks → lose context, many chunks needed
- Too large chunks → dilute the signal, exceed LLM context window
- Overlap → ensures no information falls between cracks

**Typical sizes:** 256-512 tokens with 50-100 token overlap.

## 11. Re-Ranking

After initial retrieval (fast, approximate), re-ranking uses a more expensive model to re-score the top-k results:

1. Retrieve top-100 with vector search (fast)
2. Re-rank top-100 with a cross-encoder (slow but accurate)
3. Return top-10 re-ranked results

Cross-encoders see query+document together, so they capture fine-grained relevance that bi-encoders miss.

## 12. Multi-Tenancy Patterns

Serving multiple users/tenants from one collection:

### Payload-Based
Add `tenant_id` to payload, filter on every query.
- **Pro:** Simple, one collection
- **Con:** No isolation, index shared

### Collection-Per-Tenant
Separate collection for each tenant.
- **Pro:** Full isolation, per-tenant config
- **Con:** Overhead per collection, harder to manage at scale

### Recommended: Payload + Payload Index
Use payload filtering with indexed `tenant_id` — near-zero overhead and good isolation.

## Key Terminology

| Term | Definition |
|------|-----------|
| **ANN** | Approximate Nearest Neighbor — fast similarity search |
| **HNSW** | Hierarchical Navigable Small World — graph-based ANN index |
| **Embedding** | Dense vector representation of unstructured data |
| **Sparse vector** | High-dimensional vector with mostly zero values (BM25/SPLADE) |
| **RRF** | Reciprocal Rank Fusion — method to combine ranked lists |
| **Payload** | Metadata stored alongside vectors in Qdrant |
| **RAG** | Retrieval-Augmented Generation |
| **Cross-encoder** | Model that jointly encodes query+document for accurate relevance |
| **Bi-encoder** | Model that independently encodes query and document (fast, less accurate) |
| **Chunking** | Splitting documents into smaller pieces for embedding |

## Common Pitfalls

1. **Not normalizing vectors** — cosine similarity assumes unit vectors; normalize if your model doesn't
2. **Chunks too large or too small** — experiment with 256-512 token chunks as a starting point
3. **Ignoring sparse search** — hybrid search almost always outperforms dense-only
4. **Not indexing filter fields** — unindexed payload filters cause full scans
5. **Over-retrieving for RAG** — too many chunks dilutes context; 3-5 is often enough
6. **Skipping evaluation** — always measure retrieval quality before optimizing
